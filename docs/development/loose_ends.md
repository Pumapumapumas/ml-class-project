# Loose Ends

Items noted during development that we deferred. Consulted before each phase boundary and before submission.

**Convention:** flat list of entries; mark the checkbox when resolved. Each entry has a date, one-line summary, what we'd do if we got to it, and whether it's blocking / important / nice-to-have. Resolved entries stay in the list (don't delete them) — the history is useful. See [`docs/standards/documentation_standard.md`](../standards/documentation_standard.md) for the full convention.

---

- [x] **2026-06-24 — `scripts/run_ocr.py` not yet retrofitted to `setup_logging`.** The logging-config retrofit (PR #3) migrated every CLI in `scripts/` from `logging.basicConfig(...)` to `src.utils.logging_config.setup_logging(...)`, EXCEPT `run_ocr.py`, which did not exist on `main` at the time. **Resolution:** retrofitted in commit `5d0569d` — `run_ocr.py` now calls `setup_logging(name="run_ocr", ...)` like the other four CLIs. *Severity: important.*

- [x] **2026-06-24 — `discover_books` is duplicated between the OCR and preprocessing CLIs.** `scripts/run_ocr.py` and `scripts/run_preprocessing.py` carry a near-identical ~38-line `discover_books` traversal helper, differing only in the recognized image extensions. **Resolution:** extracted to `src/utils/fs_walk.py` as `discover_books(input_root, image_extensions)` in commit `5d0569d`, with 10 unit tests in `src/utils/tests/test_fs_walk.py`. Both CLIs now call the shared helper. *Severity: nice-to-have.*

- [x] **2026-06-24 — PR #3 broke caplog assertions in test_preprocessing_e2e.** The merged logging-config retrofit (PR #3) cleared every root handler in `_reset_root_handlers`, including pytest's caplog capture handler — `test_cli_warns_and_does_not_overwrite_on_output_collision` failed because its warning assertion silently captured nothing while the warning was correctly written to stderr. **Resolution:** in commit `5d0569d`, `setup_logging` now tags its own handlers with a `_PIPELINE_HANDLER_MARKER` sentinel; the reset only removes those, leaving foreign handlers (caplog, debuggers) intact. Added `test_foreign_handlers_preserved` regression test. *Severity: important* (test was previously green on main; this restored that.)

- [ ] **2026-06-24 — `ClaudeAdapter` does not retry `anthropic.APIConnectionError` / `APITimeoutError`.** The Claude OCR adapter's retry loop catches `RateLimitError` and `APIStatusError` (status >= 500) only, exactly as the adapter task specified. Connection drops and read timeouts are *not* `APIStatusError` subclasses, so a transient network blip during a ~29 s call becomes an unretried per-page failure (the batch still continues and records it, but a network hiccup loses that page). The Gemini adapter's `ServiceUnavailable` already covers its equivalent network case, so the two adapters differ in transient-network resilience. **What we'd do:** add a third `except anthropic.APIConnectionError` branch routed through `_sleep_or_raise` (caught after `RateLimitError`/`APIStatusError`). Deferred because the adapter task explicitly enumerated the retryable set; broadening it is a scope/behavior decision for the human (Eric runs the real eval). *Severity: important* (raised by code-reviewer + quality-control in the adapter PR review).

- [ ] **2026-06-24 — Seven OCR-adapter constants/helpers are duplicated across `gemini.py` and `claude.py`.** `SYSTEM_PROMPT`, `MAX_ATTEMPTS`, `TELUGU_BLOCK_START`, `TELUGU_BLOCK_END`, `REFUSAL_MAX_CHARS`, `_backoff_delay`, `_has_telugu`, and `_looks_like_refusal` are byte-identical in both adapters. With only two adapters and the project's "three similar lines over premature abstraction" rule, duplication is currently the right call — and hoisting now would require editing the out-of-scope, load-bearing `gemini.py`. **What we'd do:** when the Tesseract adapter (Task 4) lands and becomes the third consumer, extract these into `src/ocr/_common.py` and update all three adapters' imports in a single commit. Risk if deferred: silent drift if someone tunes `REFUSAL_MAX_CHARS` or `SYSTEM_PROMPT` in one file only. *Severity: nice-to-have* (raised by refactoring-evaluator in the adapter PR review).

- [x] **2026-06-09 — Upstream dataset has dot-prefixed book directories.** The HuggingFace dataset `AlbertoChestnut/telugu-ocr` contains at least one book directory beginning with a leading `.` (e.g., `.Kalidasa-Charitra_by_chilakamarthi_lakshminarasimham`), which causes the directory to behave as a hidden file on Unix and breaks default-glob file walking. **Resolution:** `scripts/download_dataset.py` strips a single leading dot from top-level book directories after download (see `normalize_book_dirs`). The full corpus has not been audited for other unusual directory names — if we later switch to the full corpus or a different revision, re-audit and extend the normalization if needed. The dot-stripping must be disclosed in the corpus characterization report so readers can map our `book_id` to the upstream name. *Severity: nice-to-have.*
